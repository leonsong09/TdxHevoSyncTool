# user.ini extern 安全同步设计

日期：2026-03-26

## 背景

`user.ini` 是通达信个性化配置核心文件，跨版本整体复制可能导致异常。当前工具已将其排除在普通导入/导出与双向对比之外，只提供单独的 `user.ini extern 合并` 功能。

现有实现的问题在于它更接近“文本拼接”，而不是“安全同步源设置到第二个通达信”。在异常输入下，现有逻辑可能漏掉首个 `extern_*` 段、重复写入新增键，或写出格式损坏的结果。

## 目标

将源 `user.ini` 中的 `extern_*` 设置安全同步到目标通达信的 `user.ini`，满足以下要求：

1. 仅处理 `extern_*` 段落。
2. 目标中已存在的键保持不变，只补充缺失键。
3. 对异常目标结构采取阻止策略，而不是静默猜测。
4. 合并后文件结构保持可读、可回滚、可继续被通达信使用。

## 非目标

1. 不整体复制 `user.ini`。
2. 不覆盖目标段中已有键的值。
3. 不重写或整理所有普通段落。
4. 不自动修复任意损坏的 `user.ini`。

## 选定方案

采用“安全同步”方案：

1. 修复 BOM 首段识别问题。
2. 保证向已有段追加键时不会破坏行结构。
3. 对源段内部重复键做去重，避免一次同步写入同名键多次。
4. 如果源或目标 `user.ini` 存在重复同名 `extern_*` 段，直接报错并阻止合并。

未采用方案：

1. 自动写入最后一个同名目标段。
原因：目标文件结构已存在歧义，继续静默写入会放大风险。

2. 自动合并目标中的重复段。
原因：这是行为性重写，超出当前功能“安全同步”的边界。

## 设计细节

### 解析

1. 读取 `user.ini` 时应兼容 UTF-8 BOM。
2. 首个节头前的内容仍保留在 `header_lines`。
3. `extern_*` 匹配保持大小写不敏感。

### 合并规则

对于每个源 `extern_*` 段：

1. 若目标不存在同名段，则追加整个段。
2. 若目标存在且仅存在一个同名段：
   - 读取目标已有键集合。
   - 按源段顺序追加缺失键。
   - 已有键不覆盖。
   - 源段内部重复键只保留第一次出现。
3. 若源或目标存在多个同名段：
   - 直接报错。
   - 不写回目标文件。

### 写回规则

1. 写回前必须先创建 `.ini.bak` 备份。
2. 当目标现有段最后一行没有换行时，追加前必须补足换行分隔。
3. 只对需要写入的目标段做最小化追加，不重排其他段顺序。

### UI 行为

1. 预览阶段如果发现源或目标存在重复同名 `extern_*` 段，应直接提示错误，不允许继续确认合并。
2. “已存在相同段落，跳过”的语义应理解为“没有缺失键需要补充”，而不是“源目标内容完全一致”。
3. 成功后仍提示 `.ini.bak` 已生成。

## 测试策略

本次修复采用 TDD，至少覆盖以下回归测试：

1. UTF-8 BOM 文件中的首个 `extern_*` 段能被识别。
2. 源或目标存在重复同名 `extern_*` 段时，预览或应用阶段报错。
3. 目标段末行无换行时，追加后的结果仍是合法换行分隔。
4. 源段内部存在重复键时，只追加一次。
5. 正常路径下，已有键不覆盖，只补缺失键。

## 影响范围

主要影响：

1. `src/core/userini_handler.py`
2. `src/ui/userini_dialog.py`
3. `tests/` 下新增 `user.ini` 合并相关测试

不影响：

1. 普通导出/导入逻辑
2. 双向对比/同步主流程
3. `user.ini` 禁止整体迁移这一产品规则

## 验收标准

满足以下条件即可认为完成：

1. 已确认的三个边界问题均有自动化测试保护。
2. 源或目标 `user.ini` 的重复 `extern_*` 段不会再被静默放行或写坏。
3. 从源通达信同步到第二个通达信时，缺失的外挂设置能被稳定补入。
4. 相关测试通过，且不影响现有测试。
